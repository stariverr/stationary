# 架构权衡

## 1. 媒介中心化向量化（不对 Post 直接向量化）

### 背景与决策

我们在 `Media` 级别（`entity_type: EntityType.MEDIA`）对搜索文档进行索引和向量化，而不是在 `Post` 级别，即使 `Post` 代表了包含媒体的顶级逻辑实体。

### 决策依据

- **检索粒度**：应用的核心 UI 是素材网格（Media Grid）。搜索结果必须精确映射到具体且可预览的图片或视频文件。
- **上下文注入**：在 AI 丰富化过程（`enrichMediaItem`）中，父级 `Post` 的标题和描述会被注入到子级 `Media` 的 `combinedFtsContent` 中再进行向量化。因此，检索媒体向量时已隐式查询了父级帖子的文本上下文。
- **避免冗余与资源开销**：对 `Post` 进行独立向量化会引入多余的 Embedding 计算和存储开销，且由于 Post 没有物理媒体文件，无法提供独立的视觉模态向量。

### 未来展望

`EntityType.POST` 在 Schema 中仍保留，以备将来产品需求演进时支持“纯文本帖子”或专用的“帖子流（Post Feed）搜索”布局。

## 2. 单文件 DASH 视频流的播放与 Segment Base 动态提取

### 背景与决策

部分第三方媒体平台在提供高清视频时，通常会采用音视频分离的 Fragmented MP4 (fMP4) 格式（常以 `.m4s` 或 `.mp4` 结尾）。由于标准的 HTML5 `<video>` 标签无法直接合并并播放两个独立的音视频流，我们需要通过 DASH 协议（即生成 `.mpd` 描述文件）并在前端使用 `dash.js` 播放器来实现多轨道音视频流在 MSE（Media Source Extensions）中的同步播放。

对于这类单一文件存储所有分片的 DASH 轨道，`dash.js` 必须获取 **Segment Base** 元数据（包含 `Initialization range` 初始化范围和 `indexRange` 索引范围），才能通过字节范围请求（Range Request）动态请求对应的音视频分片。

### 决策依据与权衡

为了获取 Segment Base，我们在后台任务处理服务中采用了**在线流式动态提取**的方案，而非依赖爬虫端提前解析或引入离线转码。

- **动态流式读取 vs 全量内存缓冲**：
  在后台 `TaskService.processMedia` 中，媒体文件流被边下载边上传至 S3。我们通过实现流拦截，仅在内存中缓存文件的**前 32KB 数据**进行 `sidx` 盒子结构解析，定位索引范围。解析完成后将缓存数据与剩余流重新组合（Reconstructed Stream）并输出。
  - **内存开销**：固定 32KB 内存占用，即使视频文件达数 GB 级别，也绝不会增加内存或产生大文件全量缓冲的 OOM 风险。
  - **网络与 I/O 开销**：完全在原有下载流的传输途中进行，零额外网络请求，零本地磁盘临时文件开销。
  - **CPU 开销**：字节偏移比对计算量极小（耗时通常小于 0.05ms），对服务性能无任何可感知的负面影响。

- **数据库元数据闭环**：
  提取出的 `segment_base` 直接闭环存储在 `MediaFile.metadata` 列中，且在 `File` 物理表同步更新确切的字节大小 `size`。这避免了在前台生成 `.mpd` 配置文件时使用硬编码 fallback 范围造成的 `416 Range Not Satisfiable` 播放错误，保证了不同编码规格和文件头长度下播放的百分百稳定。

- **渐进增强与静默退回**：
  若下载的媒体为非 fMP4 格式（如常规 progressive 播放的单文件 MP4、封面 JPG、WebVTT 字幕等），解析器在读完 32KB 并确认无 `sidx` 盒子后，会安全地忽略并返回，对原有上传逻辑无任何副作用。
