# 系统架构设计取舍与 Trade-offs

本篇文档主要详述 Stationary 项目在开发过程中的技术决策、系统架构折中方案以及多媒体处理策略的选取考量。

---

## 1. 多轨道媒体表达模型 (Track 与 File 的拆分设计)

### 背景与决策
多媒体资源（尤其是源自社交平台的资源）在物理实体构成上常常极其复杂。例如，Bilibili 等平台的音视频轨通常是分离的（fMP4 分片）；苹果的 Live Photo（实况照片）由一张静态封面图和一段短视频共同组成；单个视频资源在播放时除了视频主音轨外，往往还需要视频封面、预生成预览图以及多国语言的字幕文件。

因此，我们摒弃了在 `Media` 表中直接定义多路 URL 列的设计（原 `primary_url`, `alternative_url`, `cover_url` 等已废弃），转而引入了**“多轨道模式”**：
- `Media` 表作为逻辑媒体资产的统一容器。
- `Track` 表（轨道变体）记录当前媒体下各个关联组件和角色的详细配置（通过 `type` 指定 `IMAGE`, `VIDEO`, `AUDIO`, `SUBTITLE`，通过 `purpose` 指定 `CONTENT`, `COVER`, `THUMBNAIL`, `PREVIEW` 等）。
- `File` 表指向存储于 S3 上的具体物理文件，以 UUID 唯一标识。

### 核心考量 (Rationales)
- **极高扩展性**：未来当需要为视频增设多路多国语言字幕、多档低码率预览流或 AI 标注生成的缩略图时，只需在 `Track` 表新增记录并绑定对应的 `File`，完全不需要对 `Media` 表结构执行物理 DDL 迁移。
- **无转码 DASH 播放支持**：系统可在不需要耗费服务器算力合并音视频轨或重新压制的前提下，通过后台同步分离的视频轨与音频轨，在播放时实时组装 MPD 清单文件挂载给 `dash.js` 播放器，达成极低延迟的统一分片流播放。
- **细粒度更新与去重**：当某路辅助轨道（如字幕或防盗链 Token 变化的音轨）需要更新时，仅需重新同步和替换单条 `Track` 及物理文件，避免了对主要高清视频资产的重复下载和存储浪费。

## 2. 单文件 DASH 视频播放与 Dynamic SegmentBase 提取

### 背景与决策
在流媒体传输规范中，为了让 `dash.js` 播放器能够直接在浏览器端同步播放分离的 fMP4 音视频轨，播放器需要在初始化阶段读取该视频轨的 `SegmentBase` 信息（即 `Initialization range` 和 `indexRange` 字节偏移量），以便以 Range 请求按需拉取分片。

为了在获取这两个字节范围的同时避免前置爬虫的解析负担，也避免将整个大视频完全缓冲写入到服务器磁盘再调用工具解析，我们在 `TaskService.processMedia` 管道中采用了一种**“流式在线提取”**的折中设计。

### 核心考量 (Rationales)
- **固定内存缓冲截留**：当数据流从源地址流式传输并写入 S3 时，我们拦截该 `ReadableStream` 并仅在内存中缓冲**前 32KB 的数据**。这段大小已足够完全解析 fMP4 的 `sidx` 容器箱，提取出分片的 `initialization` 与 `index_range` 偏移配置，并写入 `Track.metadata.segment_base`。随后，拼接缓存重新还原完整的数据流写入 S3。
- **零磁盘 I/O & 低内存消耗**：这套逻辑使得哪怕是几吉字节的大视频，其内存占用也锁定在 32KB 级别，彻底免去了写临时磁盘文件或全量内存缓冲带来的 OOM 风险。
- **高性能表现**：提取和解析偏移只需不到 0.05 毫秒的 CPU 耗时，几乎为零负担。
- **静默退化与兼容**：对于普通的 progressive MP4 或 JPEG 图片，解析器在读满 32KB 确认不存在 `sidx` 结构后会安全终止并继续正常的管道传输，保障了多格式处理的鲁棒性。

## 3. 字幕 WebVTT 格式自动转换

### 背景与决策
很多国内视频平台（例如 Bilibili）返回的字幕是其专有的 JSON 格式。这种格式无法直接在浏览器的 `<track>` 标签中被 HTML5 视频播放器读取播放。

为此，我们在 `TaskService.processMedia` 的下载过程中增加了一个拦截阶段：当检测到待下载的 `Track` 类型为 `SUBTITLE` 且格式为 `json` 时，后端会自动读取字幕正文，并在内存中将其重组为标准的 **WebVTT 格式 (`.vtt`)**，最后再以 `.vtt` 为扩展名上传至 S3。

### 核心考量 (Rationales)
- **浏览器原生兼容**：转换后的 `.vtt` 可以被播放器组件在前端无需引入额外 JSON 解析库的情况下以标准方式流畅挂载与渲染。
- **零临时磁盘开销**：解析与转换过程完全在内存中完成，流式写入，不占用任何临时文件读写周期。

## 4. 媒体主体向量化 (不执行 Post 维度直接向量化)

### 背景与决策
我们在 `Media` 维度上（`entity_type: EntityType.MEDIA`）为搜索引擎计算并持久化向量嵌入，而不是在 `Post` 维度，尽管 `Post` 才是包含媒体的逻辑实体主体。

### 核心考量 (Rationales)
- **检索粒度精准度**：Stationary 最终呈现给用户的“资产视图 (All Pins)”是纯图片/视频级别的 Masonry 布局。当搜索触发时，结果必须对应到可以直接预览或下载的具体物理图片或视频。
- **上下文信息注入 (Context Injection)**：在 AI 富化执行 `enrichMediaItem` 时，系统在生成 Media 描述时会自动注入父级 `Post` 的标题与描述。因此，在对 `Media` 的向量表示进行相似度查询时，隐式地包含了对父级帖子上下文的模糊检索能力。
- **去重与空间开销**：如果针对 `Post` 额外做一套向量计算，在实际资产列表里没有物理媒体承载的情况下会产生数据冗余，徒增计算费用和向量库存储开销。

## 5. 四级清晰度架构与转码规范

### 核心决策：四级清晰度模式
为适配不同的展示布局（大屏高清弹窗、网格视图加载、弱网移动端流畅播放以及 AI OCR/识图处理），系统建立了标准的 **四级清晰度模式**：

| 级别 | 角色名称 (Role) | 视频规范 | 图像规范 | 核心场景与目的 |
| :--- | :--- | :--- | :--- | :--- |
| **Tier 1** | **Original / Master** | 原始分辨率视频 | 原始图像文件 | **无损归档与主下载。** 不强求浏览器兼容性，用于源文件保存。 |
| **Tier 2** | **HD / Preview** | 1080p H.264 MP4 | 1600px 宽度 AVIF / WebP | **详情弹窗与大图查看主展示。** 平衡画质与加载速度。 |
| **Tier 3** | **LD / Mobile Stream** | 480p H.264 MP4 | 800px 宽度 AVIF / WebP | **移动端默认播放与预览。** 弱网或蜂窝网络极速加载。 |
| **Tier 4** | **Thumbnail / Cover** | 首秒截帧 AVIF/WebP | 400px 宽度 AVIF / WebP | **看板封面与网格流缩略图。** 极限压缩以实现数百卡片瞬时加载。 |

---

### 实现方案折中：双轨选择

#### 方案 A：CDN 动态按需裁剪 (云端/线上部署默认)
* **存储 footprint**：S3 中只保存 **Tier 1 (Original)** 以及为视频异步生成的 **Tier 4 视频封面**。
* **清晰度生成**：对于图片，前端请求原图地址并拼接变换参数（如 Cloudflare Image Resizing 路径），由 CDN 边缘节点实时缩放、变换格式并进行边缘缓存，极大节省存储成本和转换耗时。

#### 方案 B：本地离线后置转码 (私有化/局域网 NAS 部署专用)
* **双级极简配置**：在局域网（LAN）或家用 NAS 设备（如群晖、威联通等低算力 CPU）中部署时，如果强行对每个视频进行多路画质转码会导致 CPU 瞬间爆满，阻碍正常导入。针对此场景，系统支持启用双级极简策略：
  1. **Tier 1: Original**：直接保存并作为查看主文件。
  2. **Tier 2: Thumbnail**：仅生成一张 **400px** 缩略图用于列表滚动防卡顿，视频则仅截取一张封面帧。全量播放时由播放器直接通过 HTTP Range 范围请求播放 Tier 1 原片，转码开销降为零。
