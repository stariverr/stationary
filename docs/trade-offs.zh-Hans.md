# 架构权衡

## 1. 媒介中心化向量化（不对 Post 直接向量化）

### 背景与决策

我们在 `Media` 级别（`entity_type: EntityType.MEDIA`）对搜索文档进行索引和向量化，而不是在 `Post` 级别，即使 `Post` 代表了包含媒体的顶级逻辑实体。

### 决策依据

- **检索粒度**：应用的核心 UI 是素材网格（Media Grid）。搜索结果必须精确映射到具体且可预览的图片或视频文件。
- **上下文注入**：在 AI 丰富化过程（`enrichMediaItem`）中，父级 `Post` 的标题和描述会被注入到子级 `Media` 的 `combinedFtsContent` 中再进行向量化。因此，检索媒体向量时已隐式查询了父级帖子的文本上下文。
- **避免冗余与资源开销**：对 `Post` 进行独立向量化会引入多余的 Embedding 计算和存储开销，且由于 Post 没有物理媒体文件，无法提供独立的视觉模态向量。

### 未来展望

`EntityType.POST` 在 Schema 中仍保留，以备将来产品需求演进时支持“纯文本帖子”或专用的“帖子流（Post Feed）搜索”布局。

## 2. 单文件 DASH 视频流的播放与 Segment Base 动态提取

### 背景与决策

部分第三方媒体平台在提供高清视频时，通常会采用音视频分离的 Fragmented MP4 (fMP4) 格式（常以 `.m4s` 或 `.mp4` 结尾）。由于标准的 HTML5 `<video>` 标签无法直接合并并播放两个独立的音视频流，我们需要通过 DASH 协议（即生成 `.mpd` 描述文件）并在前端使用 `dash.js` 播放器来实现多轨道音视频流在 MSE（Media Source Extensions）中的同步播放。

对于这类单一文件存储所有分片的 DASH 轨道，`dash.js` 必须获取 **Segment Base** 元数据（包含 `Initialization range` 初始化范围和 `indexRange` 索引范围），才能通过字节范围请求（Range Request）动态请求对应的音视频分片。

### 决策依据与权衡

为了获取 Segment Base，我们在后台任务处理服务中采用了**在线流式动态提取**的方案，而非依赖爬虫端提前解析或引入离线转码。

- **动态流式读取 vs 全量内存缓冲**：
  在后台 `TaskService.processMedia` 中，媒体文件流被边下载边上传至 S3。我们通过实现流拦截，仅在内存中缓存文件的**前 32KB 数据**进行 `sidx` 盒子结构解析，定位索引范围。解析完成后将缓存数据与剩余流重新组合（Reconstructed Stream）并输出。
    - **内存开销**：固定 32KB 内存占用，即使视频文件达数 GB 级别，也绝不会增加内存或产生大文件全量缓冲的 OOM 风险。
    - **网络与 I/O 开销**：完全在原有下载流的传输途中进行，零额外网络请求，零本地磁盘临时文件开销。
    - **CPU 开销**：字节偏移比对计算量极小（耗时通常小于 0.05ms），对服务性能无任何可感知的负面影响。

- **数据库元数据闭环**：
  提取出的 `segment_base` 直接闭环存储在 `MediaFile.metadata` 列中，且在 `File` 物理表同步更新确切的字节大小 `size`。这避免了在前台生成 `.mpd` 配置文件时使用硬编码 fallback 范围造成的 `416 Range Not Satisfiable` 播放错误，保证了不同编码规格和文件头长度下播放的百分百稳定。

- **渐进增强与静默退回**：
  若下载的媒体为非 fMP4 格式（如常规 progressive 播放的单文件 MP4、封面 JPG、WebVTT 字幕等），解析器在读完 32KB 并确认无 `sidx` 盒子后，会安全地忽略并返回，对原有上传逻辑无任何副作用。

## 3. 图片与视频的四级清晰度保留与转码规范

### 核心决策：四级清晰度架构

为了适配 Stationary 复杂的展示版式（移动端小屏幕、PC大图灯箱、网络带宽抖动、AI富文本分析及高保真下载），平台统一规划了**四级清晰度角色**。无论是采取“服务器端离线后期转码静态存储”，还是采用“CDN边缘实时计算裁剪”，都必须按照以下四级进行设计与保留：

| 级别 (Tier) | 业务名称 (Role Name) | 视频规格配置 (Video Spec) | 图片规格配置 (Image Spec) | 核心应用场景与目的 |
| :--- | :--- | :--- | :--- | :--- |
| **第一级** | **原片 / 原始媒体**<br>(Original / Master) | 抓取源原始视频<br>(如 4K/HEVC, 音视频分离 fMP4) | 原始照片文件<br>(如 4K JPG/PNG, 原始 HEIC) | **无损归档与源文件下载**。不强求在所有客户端浏览器上可播/可看（如 ProRes 或 RAW 格式），仅作完整无损备份。 |
| **第二级** | **高清 / 大图预览**<br>(HD / Preview) | **1080p (1920x1080)**<br>H.264 MP4 (AAC 音频) | **1600px 宽度等比**<br>AVIF / WebP / JPEG | **详情页/大图灯箱的主力播放区**。兼顾极佳的视觉保真度和现代网络传输速度。100% 浏览器兼容。 |
| **第三级** | **低清 / 移动流畅**<br>(LD / Mobile Stream) | **480p (854x480)**<br>H.264 MP4 (AAC 音频) | **800px 宽度等比**<br>AVIF / WebP | **移动端默认播放/大图预览**、弱网（4G/3G）环境。保证极低延迟秒开、节省移动端用户流量，小屏幕下画质刚好够用。 |
| **第四级** | **缩略图 / 首帧图**<br>(Thumbnail / Cover) | **视频第1秒截图**<br>AVIF / WebP (400px 等比) | **320px ~ 400px 宽度**<br>AVIF / WebP (智能裁剪) | **看板视图卡片封面、媒体平铺网格**。极限压缩（单图 <20KB，视频封面对齐），支撑上百张资产卡片在单页瞬间加载完成。 |

---

### 两种技术路径的实施权衡

针对上述四级清晰度的具体落地，我们有以下两套技术实现路径，项目在不同环境下可灵活切换：

#### 路径一：CDN 实时/动态裁剪方案（当前项目主要采用的方案）

* **存储实现**：S3 中在物理上仅存储 **第一级（原片）**，视频则额外多存储一个异步截取的 **第四级（视频封面图）**。
* **清晰度衍生方式**：
  - **图片（高清/低清/缩略图）**：前端直接加载原图 URL，但通过 Cloudflare Image Resizing 携带参数（如 `/cdn-cgi/image/width=400,format=avif/...`），由 CDN 边缘节点实时缩放、转换格式（AVIF/WebP）并动态缓存。
  - **视频（高清轨/低清轨）**：直接以 HTTP Range 请求拉取原片，由客户端浏览器动态拖拽加载；在网络极差或移动端下，通过 `ALTERNATIVE` 字段指向采集源已有的低清流（如抓取到的 480p 流）以静默降级。
* **优缺点**：
  - **优点**：S3 存储开销最小（无冗余复制），开发最灵活（改前端参数即可调整尺寸，无需历史重构），导入和同步无转码等待排队。
  - **缺点**：本地开发环境或未配置付费 Cloudflare CDN 的场景会回退至直接加载原图，对 HEIC 图片或超大无损视频在没有高级 CDN 代理时加载困难。

#### 路径二：后台异步后期转码方案（完全自主可控/私有部署方案）

* **存储实现**：后台建立以 Redis/QStash 驱动的异步 `sharp` 与 `ffmpeg` 任务队列。每一个新资产入库，均在后台物理转码生成 **4 个独立的物理文件**，完整存储在 S3 中，并在 `MediaFile` 数据库表中声明不同的 `role` (如 `PRIMARY` 指向原图，`PREVIEW` 指向高清，`ALTERNATIVE` 指向低清/标清播放轨，`THUMBNAIL` 对应缩略图)。
* **清晰度衍生方式**：
  - **图片**：使用后台 `sharp` 库预先将原图生成 1600px（高清）和 400px（缩略图）两套物理 WebP/AVIF 文件并持久化。
  - **视频**：使用后台 `ffmpeg` 异步对非标准原片转码为 1080p MP4（高清）和 480p MP4（低清），开启 `-movflags +faststart` 将索引前置，以便 Range 流式秒开。
* **优缺点**：
  - **优点**：完全摆脱了第三方高级 CDN 厂商的绑定，能适应 100% 的私有化部署及局域网/内网环境；彻底在后台预处理好 HEIC 或超大视频格式，兼容性最稳健。
  - **缺点**：引入了较重的服务器端 CPU/GPU 转码负载；S3 存储因物理多份备份而有所膨胀；若未来前端卡片布局变动，需对历史数据批量跑转码脚本重构。

### 自部署与本地局域网（NAS）环境下的变体简化策略（双档位配置）

在自部署（Self-Hosted）或本地局域网（例如家用 NAS、个人工作站）场景中，网络与硬件环境发生了根本性变化，因此清晰度变体数量必须支持**可调节与动态简化**。

#### 1. 为什么自部署/局域网可以简化为“双档位”？
- **网络带宽极大**：在千兆/万兆局域网内，网络传输不再是瓶颈，加载几百 MB 的视频或几十 MB 的 RAW 照片原片基本是瞬时完成，消除了外网移动端对高昂蜂窝流量和低宽带抖动的顾虑。
- **计算资源受限**：家用 NAS（如群晖、威联通、树莓派等）的 CPU 性能非常有限，缺少专业的 GPU 硬件加速。如果强行对每个导入的文件生成四套变体（尤其是多档位视频转码），会导致 CPU 长期 100% 满载，严重影响系统响应，甚至导致导入排队数小时。
- **存储敏感度高**：部分用户希望原汁原味地保留源文件，不想因为预览用途而在本地磁盘上膨胀出多倍的重复数据。

#### 2. 推荐的“双档位”极简配置
针对自部署场景，平台建议通过环境变量（如 `MEDIA_VARIANTS_PROFILE=minimal`）将四级清晰度自动缩减为**两级**：

1. **第一档：原片 (Original)**：用于下载和主要的无损查看。
2. **第二档：低清/缩略图 (Preview/Thumbnail)**：
   - **图片**：仅预生成一套 **400px** 的小图，专门用于网格列表的快速平铺滚动，防止网页卡顿。大图详情页直接加载原图。
   - **视频**：仅提取一帧作为 **封面图** 并在列表渲染；大图播放时直接通过 HTTP Range 拖拽播放原片，跳过高清、低清预览轨的转码，将后台计算量降至最低。

通过这种“原片 + 低清缩略”的双档位设计，既利用了局域网高带宽的优势直接加载原片，又完美规避了低配服务器/NAS 在后台视频转码上的巨大算力短板。
