# Multi-Space Hybrid Search MVP

> [简体中文](./ai_hybrid_search.zh-Hans.md)

This document records the planned MVP architecture for AI-enriched search in Stationary. It is a design and implementation guide; some components may be introduced incrementally.

---

## 1. Goals

Stationary stores posts and media assets from multiple platforms. Search must support both precise textual retrieval and visual discovery:

- Exact or fuzzy keyword search over titles, descriptions, tags, authors, OCR, and AI-generated captions.
- Text semantic search over composed post/media documents.
- Visual or multimodal semantic search over image content, color, layout, style, and mood.
- Strict Library-level access control before retrieval and rank fusion.
- Mathematically safe handling of embeddings from different models and pipelines.

The MVP uses three complementary retrieval signals:

```text
keyword match
+ text semantic vector search
+ visual/multimodal vector search
→ Reciprocal Rank Fusion (RRF)
→ deduplicated entity results
```

---

## 2. Core Principles

### 2.1 Embedding Space Segregation

Vectors generated by different embedding models, model versions, task types, dimensions, preprocessing rules, or normalization policies belong to different semantic spaces. Cosine or distance comparisons across different spaces are invalid even if the physical vector dimensions match.

Therefore every embedding row must carry an `embedding_space_id` that represents the full pipeline configuration, for example:

```text
gemini:text-embedding-004:768:cosine:v1
gemini:multimodal-embedding-004:1408:cosine:v1
```

The version suffix is a **pipeline version**, not just a model version. Bump it whenever any of the following changes:

- Embedding model or model version.
- Output dimension.
- Query/document task type.
- Prompt structure or metadata composition.
- Text normalization, chunking, language handling, or tag formatting.
- Image preprocessing, resizing, frame selection, or crop policy.
- Vector normalization or distance metric.

### 2.2 Canonical Lifecycle Integration

AI-derived records mirror existing lifecycle fields:

- `delete_status`
- `delete_time`
- `recycle_time` where applicable

However, search correctness must still be evaluated against canonical `Post` and `Media` records. A search result is valid only when the canonical entity satisfies:

```sql
delete_status = 'ACTIVE'
AND recycle_time IS NULL
```

AI tables may keep denormalized lifecycle fields for indexing and operational cleanup, but they must not become the sole source of truth for visibility.

### 2.3 Pre-Filter Security Bounds

All retrieval paths must enforce Library access boundaries before expensive or ranking-sensitive operations:

1. Resolve the set of libraries visible to the current user.
2. Apply that boundary in keyword search, text semantic search, and visual semantic search.
3. Join canonical `Post` / `Media` rows and re-check `library_id`, `delete_status`, and `recycle_time`.
4. Only then perform rank fusion and return results.

RRF must never be used to merge unauthorized candidates and filter them afterward.

## 3. Input Sources vs. AI Calls

The search enrichment pipeline uses only two kinds of source inputs:

1. Raw image bytes.
2. Composed descriptive text.

To extract maximal value from those sources, the MVP performs three categories of AI calls:

| Channel                       | Input         | Model family               | Output                                             | Purpose                                                                                     |
| :---------------------------- | :------------ | :------------------------- | :------------------------------------------------- | :------------------------------------------------------------------------------------------ |
| Image Understanding           | Image bytes   | Gemini Flash               | Caption, tags, objects, colors, styles, scene, OCR | Convert visual content into structured human-readable metadata.                             |
| Visual / Multimodal Embedding | Image bytes   | `multimodal-embedding-004` | 1408-dimensional vector                            | Search by visual similarity, mood, color, composition, and cross-modal visual descriptions. |
| Text Semantic Embedding       | Composed text | `text-embedding-004`       | 768-dimensional vector                             | Search over post text, AI captions, OCR, tags, authors, concepts, and named entities.       |

This distinction is important:

- The system has two source modalities: image and text.
- It has three AI-derived retrieval/enrichment signals: VLM metadata, visual embeddings, and text embeddings.

### Example

A Xiaohongshu post contains:

- Image: a latte with heart-shaped latte art on a warm wooden table.
- Text: “I found a very healing coffee near the Seine, highly recommend this Left Bank cafe.”

Search behavior:

- Query `Seine` or `Left Bank cafe`: keyword and text semantic channels should dominate because the terms are textual concepts or named entities.
- Query `warm Japanese-style coffee material`: visual/multimodal search should contribute strongly because the request describes mood, color, and composition.
- Query `coffee latte art`: all three channels may contribute.

Note: multimodal embedding spaces can accept text queries and image documents in the same space. They are not “pixel-only” models. Their strength is mapping visual-language descriptions to image content. They should not replace keyword or text semantic retrieval for exact entities, source text, author names, or business metadata.

## 4. Planned Data Model

The MVP introduces the following logical tables in `apps/server/src/db/schema/index.ts`.

### 4.1 `embedding_space`

Registry of active, deprecated, and experimental semantic spaces.

Important fields:

- `id`: full pipeline identifier, e.g. `gemini:multimodal-embedding-004:1408:cosine:v1`.
- `provider`: embedding provider, e.g. `gemini`.
- `model` and `model_version`.
- `dimension`.
- `space_kind`: `TEXT` or `MULTIMODAL`.
- `metric`: `COSINE`, `INNER_PRODUCT`, or `L2`.
- `status`: `ACTIVE`, `DEPRECATED`, or `EXPERIMENTAL`.
- `config`: JSON configuration including supported modalities, query/document task types, normalization, preprocessing, and pipeline notes.

### 4.2 `asset_search_document`

A denormalized search document for posts and media.

It aggregates searchable text such as:

- title
- description/content
- source platform
- tags
- AI tags
- author name
- OCR text and captions from AI metadata when available

Recommended uniqueness:

```text
(entity_type, entity_id)
```

### 4.3 `asset_ai_metadata`

Structured AI analysis results for an asset.

Typical fields:

- caption
- summary
- tags
- objects
- colors
- styles
- scene
- OCR text
- model
- metadata pipeline ID
- processing status
- last error

The metadata pipeline should be versioned independently from the raw model name. A recommended ID shape is:

```text
gemini:gemini-2.5-flash:image-metadata:v1
```

Recommended uniqueness:

```text
(entity_type, entity_id, metadata_pipeline_id)
```

### 4.4 `asset_embedding`

Stores generated vectors for posts and media.

Important fields:

- `entity_type`, `entity_id`.
- `library_id` for prefiltering.
- `document_id` when tied to `asset_search_document`.
- `embedding_space_id`.
- `embedding_role`, e.g. `CONTENT_TEXT`, `AI_CAPTION`, `IMAGE_PRIMARY`, `VIDEO_KEYFRAME`, `OCR_TEXT`.
- `input_modality`, e.g. `TEXT`, `IMAGE`, `VIDEO_FRAME`, `AUDIO`.
- `dimension`.
- `embedding` using unconstrained pgvector `vector` type.
- `content_hash` for idempotency.
- `source_file_id` for file-backed media embeddings.
- `embedding_status`, e.g. `READY`, `STALE`, `FAILED`, `DISABLED`.
- lifecycle fields.

Recommended uniqueness:

```text
(entity_type, entity_id, embedding_space_id, embedding_role, content_hash)
```

`content_hash` should be non-null or the unique index should use `NULLS NOT DISTINCT`; otherwise PostgreSQL allows duplicate rows with `NULL` hash values.

## 5. Vector Storage and Indexing

### 5.1 Generic Vector Column

Use a single unconstrained pgvector column:

```sql
embedding vector NOT NULL
```

Store the actual dimension in `dimension` and validate it:

```sql
ALTER TABLE asset_embedding
ADD CONSTRAINT check_vector_dims
CHECK (vector_dims(embedding) = dimension);
```

Application code must also validate provider output dimensions before insert. Database constraints are the last line of defense, not the primary validation mechanism.

### 5.2 Partial HNSW Indexes

Create model-specific partial HNSW expression indexes:

```sql
CREATE INDEX IF NOT EXISTS asset_embedding_gemini_mm_1408_hnsw
ON asset_embedding
USING hnsw ((embedding::vector(1408)) vector_cosine_ops)
WHERE embedding_space_id = 'gemini:multimodal-embedding-004:1408:cosine:v1'
  AND dimension = 1408
  AND embedding_status = 'READY'
  AND delete_status = 'ACTIVE';

CREATE INDEX IF NOT EXISTS asset_embedding_gemini_text_768_hnsw
ON asset_embedding
USING hnsw ((embedding::vector(768)) vector_cosine_ops)
WHERE embedding_space_id = 'gemini:text-embedding-004:768:cosine:v1'
  AND dimension = 768
  AND embedding_status = 'READY'
  AND delete_status = 'ACTIVE';
```

Query predicates should match these partial-index predicates as closely as possible.

### 5.3 Extensions and Migration Notes

Required PostgreSQL extensions:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

Operational notes:

- `CREATE INDEX CONCURRENTLY` must run outside transaction blocks.
- Extension initialization should be handled as a dedicated bootstrap step because managed databases may require elevated permissions or extension allow-listing.
- Development can use normal `CREATE INDEX`; production should prefer `CREATE INDEX CONCURRENTLY` for large tables.
- Drizzle-generated DDL may need manual companion migrations for extensions, pgvector HNSW indexes, and trigram indexes.

### 5.4 Keyword Indexes

Use `pg_trgm` indexes for fuzzy keyword retrieval:

```sql
CREATE INDEX IF NOT EXISTS asset_search_doc_title_trgm_idx
ON asset_search_document
USING gin (title gin_trgm_ops);

CREATE INDEX IF NOT EXISTS asset_search_doc_content_trgm_idx
ON asset_search_document
USING gin (content gin_trgm_ops);
```

Also maintain B-tree indexes for filtering:

```sql
CREATE INDEX IF NOT EXISTS asset_search_doc_library_status_idx
ON asset_search_document (library_id, delete_status, entity_type, entity_id);

CREATE INDEX IF NOT EXISTS asset_embedding_space_library_idx
ON asset_embedding (embedding_space_id, library_id);

CREATE INDEX IF NOT EXISTS asset_embedding_entity_idx
ON asset_embedding (entity_type, entity_id);
```

## 6. Provider Abstraction

The AI provider should explicitly model query/document purpose for text embeddings:

```ts
export interface ImageAiMetadata {
    caption: string;
    tags: string[];
    objects: string[];
    colors: string[];
    styles: string[];
    scene: string;
    ocrText: string;
}

export interface EmbedTextParams {
    text: string;
    spaceId: string;
    purpose: "QUERY" | "DOCUMENT";
}

export interface EmbedImageParams {
    imageBuffer: Buffer;
    mimeType: string;
    spaceId: string;
}

export interface AiProvider {
    embedText(params: EmbedTextParams): Promise<number[]>;
    embedImage(params: EmbedImageParams): Promise<number[]>;
    describeImage(imageBuffer: Buffer, mimeType: string): Promise<ImageAiMetadata>;
}
```

Implementation requirements:

- Route by `spaceId` and reject unknown spaces.
- Use task types from `embedding_space.config`.
- Validate output dimension immediately.
- Keep API keys in environment variables only.
- Use schema validation for structured image metadata responses.
- Make generation idempotent using `content_hash` and pipeline IDs.
- Tests should mock providers rather than calling external AI APIs by default.

## 7. Search Flow

A high-level MVP search flow:

```ts
async function search(params: SearchParams, userId: string) {
    const accessibleLibraryIds = await resolveAccessibleLibraries(userId, params.library_id);
    if (accessibleLibraryIds.length === 0) {
        return { list: [], total: 0 };
    }

    const textSpace = "gemini:text-embedding-004:768:cosine:v1";
    const visualSpace = "gemini:multimodal-embedding-004:1408:cosine:v1";

    const [keywordResults, textSemanticResults, visualSemanticResults] = await Promise.all([
        keywordSearch(params, accessibleLibraryIds),
        params.q ? textSemanticSearch(params.q, textSpace, accessibleLibraryIds) : [],
        params.q ? visualSemanticSearch(params.q, visualSpace, accessibleLibraryIds) : [],
    ]);

    return reciprocalRankFusion([keywordResults, textSemanticResults, visualSemanticResults], {
        k: 60,
        weights: [1.0, 1.0, 1.2],
        dedupeKey: (item) => `${item.entity_type}:${item.entity_id}`,
    });
}
```

Important details:

- Keyword search should be strongest for exact names, source text, author names, brands, and platform-specific identifiers.
- Text semantic search should cover paraphrases and conceptual matches over composed documents.
- Visual/multimodal search should cover style, mood, color, composition, and visual-language descriptions.
- Vector channels should retrieve more candidates than the final page size before RRF.
- `total` in hybrid search is not necessarily an exact database-wide total; prefer documenting it as candidate count or returning `hasMore` if precise pagination is not implemented.

## 8. Lifecycle and Sync Rules

When deleting a `Post` or `Media`, update AI-derived tables in the same transaction where practical:

```text
asset_search_document.delete_status = 'DELETED'
asset_ai_metadata.delete_status = 'DELETED'
asset_embedding.delete_status = 'DELETED'
```

Also set `delete_time`.

For recycle/restore and library movement:

- Search must rely on canonical joins to `Post` / `Media` for final visibility.
- Derived `library_id` fields should be synchronized when entities move libraries.
- Stale derived rows must not cause access leaks because canonical `library_id` and lifecycle checks are always reapplied.

## 9. Verification Checklist

Automated checks:

- Schema migrations create required tables and constraints.
- pgvector dimension mismatch fails fast.
- Provider output dimension validation throws before database insertion.
- Unknown `embedding_space_id` is rejected.
- RRF deduplicates by `entity_type:entity_id`.
- Library access filtering applies to keyword, text semantic, and visual semantic paths.
- Recycled or deleted `Post` / `Media` rows do not appear in search.

Manual checks:

1. Seed `embedding_space` rows for active text and multimodal spaces.
2. Process representative images such as coffee photos, dark technology graphics, screenshots with OCR text, and multi-image posts.
3. Verify generated metadata, text embeddings, and image embeddings have expected dimensions.
4. Run natural language searches across restricted libraries and verify no unauthorized results appear.
5. Inspect matched channels and rankings for keyword-heavy, concept-heavy, and visual-style-heavy queries.

---

## 10. MVP Non-Goals

The first implementation does not need to solve every future retrieval problem. The following can be deferred:

- Full video understanding beyond selected keyframes.
- Audio embeddings.
- Cross-model migration tooling.
- Exact global result counts for hybrid search.
- Sophisticated learning-to-rank models.
- Automated quality evaluation dashboards.

The MVP should prioritize correctness, safe access boundaries, idempotent AI processing, and clear separation of embedding spaces.
